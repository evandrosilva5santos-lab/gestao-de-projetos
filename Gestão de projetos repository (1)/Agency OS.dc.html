DOCUMENTAÇÃO DO PRODUTO

Agency OS

Plataforma de processamento e distribuição de leads de uma agência (multi-cliente). O objetivo é substituir as automações frágeis do n8n por um sistema único, confiável e auditável. Arquivo completo no projeto: DOCUMENTACAO.md.

1. Arquitetura

O coração é o Motor de Processamento de Leads. Todos os outros módulos (CRM, Clientes, Regras, Dashboard) são consumidores dele. Integrações externas (Meta, Google, Kommo, Sheets, Evolution/WhatsApp) são apenas adapters/providers — nunca contêm regra de negócio.

Origem → Motor (toda a regra) → CRM · Sheets · WhatsApp · APIs

2. Princípios

  • Fluxo único — todo lead, de qualquer origem, passa pelo mesmo pipeline.
  • Idempotência — cada Lead ID é processado uma vez só (sem duplicar lead, mensagem, CRM ou planilha).
  • Banco é a fonte oficial — Sheets vira destino opcional; se cair, o lead continua salvo.
  • Reprocesso por etapa — só a etapa que falhou é refeita.
  • Distribuição transacional — a Rodada da Vez vive no sistema, com lock de concorrência.

3. Módulos

  • Motor de Processamento — pipeline único, trace por etapa, idempotência, modelo interno e camada de adapters.
  • Fontes de Entrada — conexão Meta/Google por token, wizard e mapeamento formulário → workspace (com config por formulário).
  • Dashboard Motor — Auditoria e Performance (destaque, funil de conversão, SLA, ranking).
  • CRM & Funil — Kanban arrastável e lista agrupável; mover status; soma por coluna.
  • Clientes (Workspaces) — cada cliente é um workspace isolado.
  • Regras de Roteamento — Kommo/Planilha, métodos de distribuição e regras por origem.
  • Vendedores & Rodada — equipe, WhatsApp (Evolution API) e Rodada da Vez interna.
  • Logs & Automação — stream de logs, auto-correção (pull) e rotas.

4. Modelo interno do lead

id_interno · id_origem (Lead ID) · origem · workspace · cliente · formulario · nome · telefone · email · campos_personalizados · payload_original · data · status · logs[]

5. Central de Integrações

Cada plataforma é cadastrada uma única vez no nível do sistema e reutilizada por N workspaces. A integração pertence ao sistema, não ao cliente — o cliente nunca insere token nem ID, apenas consome uma conexão existente (Provedor → Conexão → Página → Formulário → Vincular).

  • Provedores: Meta, Google, Google Sheets, Kommo, Evolution API (extensível).
  • Cards com status (Conectado / Token expirado com motivo exato / Sincronizando), token mascarado e contadores de formulários · páginas · workspaces.
  • Ações reativas: Testar · Sincronizar · Renovar token · Desconectar/Reconectar.
  • Wizard "Nova integração" (Provedor → Autenticação → árvore Business ▸ Página ▸ Formulários).

6. Integração Meta (Graph API) — por perfil

  • Gestor / agência (uso principal): Token permanente (System User) — configura uma vez, não expira, sem login/2FA recorrente.
  • Cliente final: OAuth "Entrar com Facebook" — 1 clique, sem token nem ID, para quem não é técnico.
  • Validação obrigatória via /debug_token (lê is_valid, expires_at, escopos) e exibição do motivo exato do erro (ex.: 190). Webhook leadgen → roteamento por Form ID → Workspace. Detalhes completos na seção 10 do DOCUMENTACAO.md.

7. Notificações (por cliente)

Dentro de cada Workspace (aba Notificações). Disparam só depois de tratar, validar, distribuir e definir o vendedor, com dados normalizados. Editor de template funcional com prévia em tempo real e placeholders clicáveis (inclui os campos personalizados do formulário do cliente).

  • Grupo (obrigatória) — registra toda entrada no grupo operacional.
  • Vendedor (opcional) — vai ao WhatsApp do vendedor da vez.
  • Cliente (opcional) — sai do WhatsApp do vendedor responsável, nunca de um número central.
  • Auditoria — hora, tipo, destinatário (com "via {vendedor}"), template e status (Enviado/Reprocessado/Falhou).

8. Ordem de execução

Receber → Normalizar → Validar duplicidade → Rodada da Vez → Definir vendedor → Salvar no banco → Integrações (CRM/Sheets) → Msg Grupo → Msg Vendedor → Msg Cliente (via WhatsApp do vendedor) → Logs

9. Estado atual

Protótipo de design interativo. As interações de UI funcionam; backend, banco, filas (Inngest), APIs reais (Graph API, Kommo, Evolution) e locks de concorrência são a próxima fase de engenharia (ver seção 8 do DOCUMENTACAO.md).
